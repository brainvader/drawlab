## Test as a Context（TaaC）— 開発哲学

### コンセプト

**Test as a Context (TaaC)** は LLM を活用したアプリ開発の中核的な哲学である。

> テストファイルを書くことが、そのままコンテキスト定義になる。
> コンテキストを定義することが、テストを書くことと等価である。

これは **Context First = Test First Development** を意味する。

### 従来の TDD との違い

|                        | TDD                    | TaaC                                   |
| ---------------------- | ---------------------- | -------------------------------------- |
| テストの目的           | 実装の正しさを証明する | コンテキスト（作業スコープ）を定義する |
| テストを書くタイミング | 実装前                 | 開発開始前（スコープ定義として）       |
| テストの副産物         | リグレッション防止     | LLM に渡すファイル群の自動特定         |
| コンテキストの定義     | 手動・暗黙的           | テストファイルから自動導出             |

### TaaC の開発フロー

```
1. ContextMap.html でUI構造とコンテキストを定義する
   → デザインカンプとして画面構成を表現する
   → 各コンポーネントに describe/it を列挙する（未実装は [ ]）

2. テストファイルを書く
   → describe 名 = 作業コンテキスト名（ユーザーの行動を軸にした日本語）
   → import 群  = 作業スコープ（関与ファイル群）
   → it         = 振る舞いの記述

3. そのスコープで LLM に実装を依頼する
   → テストファイルを渡すだけで LLM がスコープを把握できる
   → トークン消費を削減しつつ文脈が明確になる

4. テストを実行して確認する
   → グリーンになったら ContextMap の [ ] を [✓] に更新する
   → 失敗ログを LLM にペーストして修正を依頼する
```

### テストファイルの構造

```ts
// import 群がスコープを定義する
import { FileTree } from './FileTree'
import { useSourceGraph } from '../hooks/useSourceGraph'

// describe 名がコンテキスト名になる（ユーザーの行動を軸にした日本語）
describe('ファイルツリーを閲覧・選択する', () => {
    it('rootPath 配下のエントリがツリー表示される', () => { ... })
    it('ディレクトリをクリックすると子エントリが展開される', () => { ... })
    it('ファイルクリックで onFileClick が呼ばれる', () => { ... })
})
```

### テストファイルが存在しない = コンテキストが未定義

TaaC において、テストファイルのないコンポーネントはコンテキストが定義されていない状態である。
これは設計上の圧力として機能する：

> 「テストを書かないとスコープが定義できない」
> → テストファイルの存在がコンテキスト定義の前提条件になる

### コンテキストの粒度

1ファイル1コンテキストを推奨する。ファイル名がコンテキスト名の代わりになる。

```ts
// FileTree.test.tsx — FileTree コンポーネントのコンテキスト
// useSourceGraph.test.ts — useSourceGraph フックのコンテキスト
// store.test.ts — Zustand ストアのコンテキスト
```

### ContextMap との関係

```
ContextMap.html    → テストの目次（describe/it を [✓]/[ ] でチェック）
Vitest テストファイル → コンテキストのSSOT（importがスコープ、describe/itが振る舞い）
実装ファイル        → テストに従って実装する
Storybook          → 視覚的な状態確認
Playwright         → 複数コンテキストをまたぐ統合シナリオ
```
